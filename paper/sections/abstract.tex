Production scientific codes read their behavior from \emph{configuration files} at startup and then never change it. These configuration values, fixed for an entire multi-day run, are invisible to optimizing compilers, because they arrive through an I/O barrier into module-level state. On a NAS~LU kernel we show the barrier is structural: an externally-linked configuration-style module global is not folded into its loop bounds at any optimization level, since the value is absent from the program and the global is not internalizable, and production configuration state is both. Even injecting the value and running whole-program LTO leaves a loop bound the code derives from it guarded rather than constant. The result is a \emph{configuration tax} paid on every timestep. We present \emph{deployment-time specialization}: we statically recover the glacial configuration variables a namelist fixes and inject them, so the standard optimizer cascade (constant propagation, dead-code elimination, fusion, vectorization) can fire. A recovered value also fixes \emph{types} and index \emph{ranges}. We lift the recovered domain under one \emph{interval-arithmetic-on-collections} lattice to a finite type-set that devirtualizes polymorphic dispatch, our central mechanism and one the standard pipeline does not perform even with the value in hand, and to integer intervals that narrow index storage. Finite-domain configurations generate a minimal covering set of pre-compiled, unguarded binaries dispatched once at startup. We evaluate on the ICON climate model, ECMWF's ecRad radiation scheme, and Quantum ESPRESSO across NVIDIA GH200 and AMD MI300A, comparing against \texttt{nvfortran -O4} and against the strongest standard-tool baseline, the recovered values injected as constants under whole-program LTO, which recovers the directly-injected scalars but, we expect, not the derived bounds, devirtualization, or index narrowing.
