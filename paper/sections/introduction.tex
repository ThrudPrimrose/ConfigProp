\section{Introduction}
\label{sec:intro}

Production scientific codes read their settings from a file. The ICON climate model~\cite{icon}, the ecRad radiation scheme~\cite{ecrad2024}, and Quantum ESPRESSO each read a Fortran \emph{namelist} once at startup and store a few hundred values in module-level variables, including \begin{enumerate*}[label=(\roman*)] \item grid and spectral sizes, \item solver choices, and \item physics switches\end{enumerate*}. None of them changes again for the rest of a run. They are written once, or rarely, and read everywhere, which makes them \emph{glacial}~\cite{glacial}. They are constants of the run, but the compiler never treats them as such. The value comes from a data file the compiler does not read, and it sits in externally-linked module state, so it stays a runtime value at every optimization level. The code keeps paying for branches, dead code, and blocked loop fusion and vectorization that the settings already decided. We call this the \emph{configuration tax}.

We remove it by recovering the config-fixed values statically and specializing the code to the deployment. No profiling is needed. Profile-guided optimization runs the program to learn which values are stable in practice; for configuration variables the file already gives them, exactly. That is a third option next to the usual two: ahead-of-time compilation does not know the deployment values, and just-in-time compilation knows them but pays to recompile at runtime. We perform \emph{deployment-time specialization}: an ahead-of-time build driven by the config file and by the analysis below, with the right binary chosen once at startup. We parse the namelist, trace each variable across the read into the module state it initializes, recover its domain, and inject it.

\paragraph{A set-of-intervals analysis, supplied by the configuration.}
At the center of the approach is a set-valued range analysis, and the configuration file is its natural supplier. Ordinary range analysis has to \emph{infer} how wide a value can be; a namelist instead \emph{states} it, so each recovered variable arrives with a tight bounded domain at no analytical cost. We carry every such domain in one lattice, \emph{interval arithmetic on collections} (\Cref{sec:approach}): a value is abstracted as a bounded set or an interval, and the abstraction is lifted from single values to collections of them, giving sets of intervals and, in the same framework, sets of types. The shape of the recovered set is what selects the optimization. A \emph{singleton} is an ordinary constant. A small \emph{finite value-set}, a boolean or a few integer modes, is specialized over the cross-product of its elements. An \emph{integer interval}, taken from the array shapes a configuration fixes, narrows index storage to the smallest integer type that spans it. A finite \emph{set of types}, the concrete classes a polymorphic selector admits, collapses dynamic dispatch: lifting the recovered value to that set turns a v-table into an \lstinline[style=fortran]|if|-chain that configuration constants prune to the one deployed target. One recovery and one fixpoint feed all four; only the residual use of the domain differs. Because the configuration guides this analysis the way a profile guides PGO, but exactly rather than statistically, we call the result \emph{deployment-guided optimization}.

The type-set case is our centerpiece. Index narrowing and devirtualization do not follow from injecting a scalar and rerunning the optimizer: the default Fortran pipeline (flang, nvfortran) does not devirtualize even from a known dynamic type (\Cref{sec:spec}), and we know of no prior specializer that derives a \emph{type}-set from a recovered configuration value to drive it (\Cref{sec:related}).

Configuration reaches a kernel through a second channel too. Besides the glacial namelist state above, a driver configures each kernel invocation with mode arguments from a small finite set, exactly the finite value-set case of the analysis. \Cref{lst:motivating} is one, a loop nest from ICON's dynamical core, lightly distilled. It runs under \texttt{istep}, the predictor/corrector stage of the dynamical core, and \texttt{lvn\_only}, a flag requesting only the normal-velocity tendency. Kernels \texttt{A}, \texttt{B}, and \texttt{C} sweep the same column of edge values and reuse the velocities \texttt{vn} and \texttt{vt}, but the compiler keeps them as three separate loops: it cannot fuse \texttt{A} with \texttt{C} across the \lstinline[style=fortran]|IF (.NOT. lvn_only)| guard, nor drop the \lstinline[style=fortran]|IF (istep == 1)| test, because both arguments are unknown to it. On bandwidth-bound code the velocities are read from memory three times instead of once. Each argument ranges over only a few values, so specializing the kernel on their cross-product and dispatching at the call site removes the barrier: \texttt{istep}\,$=$\,2 makes the block dead; \texttt{istep}\,$=$\,1 with \texttt{lvn\_only}\,$=$\,\texttt{.true.} drops \texttt{B} and fuses \texttt{A} and \texttt{C}; \texttt{lvn\_only}\,$=$\,\texttt{.false.} makes \texttt{B} unconditional and fuses all three.

\begin{lstlisting}[style=fortran,basicstyle=\ttfamily\footnotesize,float=t,caption={A configuration-dependent fusion barrier in ICON's \texttt{velocity\_tendencies} (distilled). The invocation arguments \texttt{istep} (dynamical-core stage) and \texttt{lvn\_only} (which velocity tendencies to compute) are fixed within a call but unknown to the compiler, which keeps kernels \texttt{A}--\texttt{C} unfused. Each ranges over a small finite set.},label={lst:motivating}]
! istep, lvn_only: per-call mode flags, each from a small finite set.
IF (istep == 1) THEN
  DO jk = 2, nlev                    ! kernel A
    vn_ie(je,jk) = w*vn + (1-w)*vn_km1
    z_kin(je,jk) = 0.5*(vn**2 + vt**2)
  END DO
  IF (.NOT. lvn_only) THEN           ! config barrier
    DO jk = 2, nlev                  ! kernel B
      z_vt(je,jk) = w*vt + (1-w)*vt_km1
    END DO
  END IF
  DO jk = 2, nlev                    ! kernel C
    z_cc(je,jk) = vn*ddxn_z + vt*ddxt_z
  END DO
END IF
\end{lstlisting}

People already do this by hand, at cost. ECMWF's recent twelvefold speedup of ecRad~\cite{ecrad2024} splits into a spectral factor and a separable, roughly threefold code-optimization factor. Hard-coding the g-point count so that loop bounds become constant, which is configuration propagation done by hand, is one of the changes behind that second factor. It takes hundreds of person-hours, has to be redone for each configuration, and breaks quietly when a setting no longer matches. Deployment-guided optimization does it automatically, and because the dimensions are known at compile time the GPU backend can emit tighter kernels: constant launch bounds, unrolled loops, narrower index types.

\paragraph{Contributions.}
\begin{itemize}
  \item We frame configuration recovery as a \emph{set-of-intervals} analysis supplied by the config file, unifying constants, finite value-sets, integer intervals, and type-sets in one lattice (\Cref{sec:approach}), and we characterize the \emph{configuration tax}: on a NAS~LU kernel the barrier is structural for two independent reasons, value-behind-I/O and external linkage, so no optimization level recovers production configuration state (\Cref{sec:background}). We position \emph{deployment-time} and \emph{deployment-guided} optimization against AOT, JIT, and PGO.
  \item \textbf{Glacial recovery across the I/O barrier} (\Cref{sec:recovery}): a static analysis that traces namelist reads to the glacial module-level state they initialize and recovers it as deployment-time constants, the value the standard pipeline is missing and structurally cannot obtain.
  \item \textbf{Type-domain devirtualization} (\Cref{sec:spec}, our centerpiece): we lift the value/interval lattice to a finite type-set. To our knowledge this is the first analysis to drive devirtualization from a configuration-recovered value domain, eliminating dispatch that the default Fortran pipeline (flang, nvfortran) does not remove even when the dynamic type is known.
  \item \textbf{Variant selection} (\Cref{sec:variants}): finite-domain cross-product specialization with greedy minimal-covering-set selection over a distribution of operational namelists, deployed as multiple \emph{unguarded} whole-program binaries dispatched once at startup.
  \item \textbf{Evaluation} (\Cref{sec:eval}) on ICON, ecRad, and Quantum ESPRESSO across GH200 and MI300A, against \texttt{nvfortran -O4} with and without profile-guided optimization\footnote{Current nvfortran (26.3) deprecates and ignores \texttt{-Mipa} and does not accept \texttt{-flto}; its only surviving interprocedural optimization is inlining, which does not propagate a namelist value held in module state, so even \texttt{-O4} folds no configuration global. Profile-guided optimization (\texttt{-Mpfi}/\texttt{-Mpfo}) does not fold it either.} and against the strongest standard-tool baseline: the recovered values injected as constants under whole-program LTO, which folds the directly-injected scalars but still leaves the derived loop bounds, devirtualization, and index narrowing that our analysis targets.
\end{itemize}

We build on the DaCe dataflow compiler and its Fortran frontend, which gives the whole-program view our specializations need without depending on the maturity of Flang/LLVM LTO on production-scale Fortran.
