\section{Introduction}
\label{sec:intro}

Large scientific codes read their settings from a file. The ICON climate model~\cite{icon}, the ecRad radiation scheme~\cite{ecrad2024}, and Quantum ESPRESSO each read a Fortran \emph{namelist} once at startup and store a few hundred values in module-level variables: grid and spectral sizes, solver choices, physics switches. None of them changes again for the rest of a run that lasts hours or days. They are written once and read everywhere, which makes them \emph{glacial}~\cite{glacial}. They are constants of the run, but the compiler never treats them as such. The value comes from a data file the compiler does not read, and it sits in externally-linked module state, so it stays a runtime value. The code keeps paying for branches, dead code, and blocked loop fusion and vectorization that the settings already decided. We call this the \emph{configuration tax}.

We remove it by recovering the config-fixed values statically and specializing the code to the deployment. No profiling is needed. Profile-guided optimization runs the program to learn which values are stable in practice; for configuration variables the file already gives them, exactly. That is a third option next to the usual two: ahead-of-time compilation does not know the deployment values, and just-in-time compilation knows them but pays to recompile at runtime. We do \emph{deployment-time specialization}: an ahead-of-time build driven by the config file, with the right binary chosen once at startup. We parse the namelist, trace each variable across the read into the module state it initializes, recover its domain, and inject it.

The compiler cannot recover these constants on its own, for two structural reasons; \Cref{sec:background} shows both on a NAS~LU kernel whose grid bounds are module variables, as in any code built from \texttt{COMMON} blocks or configuration modules. First, the value is not in the program. It enters through an \emph{I/O barrier}, a namelist read that lowers to opaque runtime calls, and constant propagation has to give up there. Second, it is module-level state, external by linkage and reached across many translation units through \lstinline[style=fortran]|USE ..., ONLY:|, so no single translation unit can assume it is unwritten. Configuration state fails both tests and stays a runtime value at every optimization level, whole-program LTO included (\Cref{tab:lu}). For a scalar, that is all that was missing: supply the value as a source-level constant and the usual passes (constant propagation, dead-code elimination, fusion) finish the job. The hard part is obtaining that constant, soundly and statically, from a file the compiler never reads.

Configuration reaches a kernel through a second channel too. Besides the glacial namelist state above, a driver configures each kernel invocation with mode arguments drawn from a small finite set, and these create the same kind of barrier. \Cref{lst:motivating} is one, a loop nest from ICON's dynamical core, lightly distilled. It runs under \texttt{istep}, the predictor/corrector stage of the dynamical core, and \texttt{lvn\_only}, a flag requesting only the normal-velocity tendency. Kernels \texttt{A}, \texttt{B}, and \texttt{C} sweep the same column of edge values and reuse the velocities \texttt{vn} and \texttt{vt}, but the compiler keeps them as three separate loops: it cannot fuse \texttt{A} with \texttt{C} across the \lstinline[style=fortran]|IF (.NOT. lvn_only)| guard, nor drop the \lstinline[style=fortran]|IF (istep == 1)| test, because both arguments are unknown to it. On bandwidth-bound code the velocities are read from memory three times instead of once. Each argument ranges over only a few values, so specializing the kernel on their cross-product and dispatching at the call site removes the barrier: \texttt{istep}\,$=$\,2 makes the block dead; \texttt{istep}\,$=$\,1 with \texttt{lvn\_only}\,$=$\,\texttt{.true.} drops \texttt{B} and fuses \texttt{A} and \texttt{C}; \texttt{lvn\_only}\,$=$\,\texttt{.false.} makes \texttt{B} unconditional and fuses all three.

The invocation arguments \texttt{istep} (dynamical-core stage) and \texttt{lvn\_only} (which velocity tendencies to compute) are fixed within a call but unknown to the compiler, which keeps kernels \texttt{A}--\texttt{C} unfused. Each ranges over a small finite set.},label={lst:motivating}]
! istep, lvn_only: per-call mode flags, each from a small finite set.
IF (istep == 1) THEN
  DO jk = 2, nlev                    ! kernel A
    vn_ie(je,jk) = w*vn + (1-w)*vn_km1
    z_kin(je,jk) = 0.5*(vn**2 + vt**2)
  END DO
  IF (.NOT. lvn_only) THEN           ! config barrier
    DO jk = 2, nlev                  ! kernel B
      z_vt(je,jk) = w*vt + (1-w)*vt_km1
    END DO
  END IF
  DO jk = 2, nlev                    ! kernel C
    z_cc(je,jk) = vn*ddxn_z + vt*ddxt_z
  END DO
END IF
\end{lstlisting}

Often the recovered domain is a set: a handful of modes, an integer interval, or a set of types. These are the cases that matter most, because the pipeline cannot finish them even with the value in hand. We handle four kinds of domain in one lattice, \emph{interval arithmetic on collections}. A \emph{singleton} folds to a constant. A small \emph{finite value-set}, such as a boolean or a few integer modes, drives specialization over the cross-product of its elements, as with \texttt{lvn\_only} and \texttt{istep} above. An \emph{integer interval}, taken from the shape an index array addresses, narrows that array to the smallest integer type that fits. A finite \emph{type-set} collapses dynamic dispatch. The type-set is the main case. A configuration variable that picks one of several linear solvers has a finite type-set; lifting the value to that set turns the v-table into an \lstinline[style=fortran]|if|-chain that configuration constants prune to the one deployed solver. Index narrowing and devirtualization do not follow from injecting a scalar and rerunning the optimizer. The default Fortran pipeline (flang, nvfortran) does not devirtualize even from a known dynamic type (\Cref{sec:spec}), and we know of no prior specializer that derives a type-set from a recovered configuration value to drive it (\Cref{sec:related}).

People already do this by hand, at cost. ECMWF's recent twelvefold speedup of ecRad~\cite{ecrad2024} splits into a spectral factor and a separable, roughly threefold code-optimization factor. Hard-coding the g-point count so that loop bounds become constant, which is configuration propagation done by hand, is one of the changes behind that second factor. It takes hundreds of person-hours, has to be redone for each configuration, and breaks quietly when a setting no longer matches. Deployment-time specialization does it automatically, and because the dimensions are known at compile time the GPU backend can emit tighter kernels: constant launch bounds, unrolled loops, narrower index types.

\paragraph{Contributions.}
\begin{itemize}
  \item We characterize the \emph{configuration tax} and show, on a NAS~LU kernel, that the barrier is structural for two independent reasons, value-behind-I/O and external linkage, so no optimization level recovers production configuration state (\Cref{sec:background}). We position \emph{deployment-time specialization} (config-file-driven AOT specialization with startup binary selection) against AOT, JIT, and PGO.
  \item \textbf{Glacial recovery across the I/O barrier} (\Cref{sec:recovery}): a static analysis that traces namelist reads to the glacial module-level state they initialize and recovers it as deployment-time constants, the value the standard pipeline is missing and structurally cannot obtain.
  \item \textbf{Type-domain devirtualization} (\Cref{sec:spec}, our centerpiece): we lift the value/interval lattice to a finite type-set. To our knowledge this is the first analysis to drive devirtualization from a configuration-recovered value domain, eliminating dispatch that the default Fortran pipeline (flang, nvfortran) does not remove even when the dynamic type is known.
  \item \textbf{Variant selection} (\Cref{sec:variants}): finite-domain cross-product specialization with greedy minimal-covering-set selection over a distribution of operational namelists, deployed as multiple \emph{unguarded} whole-program binaries dispatched once at startup.
  \item \textbf{Evaluation} (\Cref{sec:eval}) on ICON, ecRad, and Quantum ESPRESSO across GH200 and MI300A, against \texttt{nvfortran -O4} with and without profile-guided optimization\footnote{Current nvfortran (26.3) deprecates and ignores \texttt{-Mipa} and does not accept \texttt{-flto}; its only surviving interprocedural optimization is inlining, which does not propagate a namelist value held in module state, so even \texttt{-O4} folds no configuration global. Profile-guided optimization (\texttt{-Mpfi}/\texttt{-Mpfo}) does not fold it either.} and against the strongest standard-tool baseline: the recovered values injected as constants under whole-program LTO, which folds the directly-injected scalars but still leaves the derived loop bounds, devirtualization, and index narrowing that our analysis targets.
\end{itemize}

We build on the DaCe dataflow compiler and its Fortran frontend, which gives the whole-program view our specializations need without depending on the maturity of Flang/LLVM LTO on production-scale Fortran.
